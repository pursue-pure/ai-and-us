# 可维护性五因素自评报告

对照教材 §8.4.1 的五个因素，对当前代码（`game/` 与 `tests/`）进行 1-5 分制自评。

## 评估口径说明

- 评分区间定义：
  - 1 分：明显不可维护，缺少基本工程约束。
  - 3 分：可维护但存在结构性短板，需要持续治理。
  - 5 分：具备较强可演进能力，质量门禁与工程规范完整。
- 评估对象：当前 `game/` 运行代码、`tests/` 单元测试、`README.md` 与 CI 流水线配置。
- 评估维度：结构清晰度、测试保护强度、改动影响面、环境一致性、跨场景复用能力。

## 评分总览

| 因素 | 分数（1-5） | 结论 |
|------|-------------|------|
| 可理解性 | 4 | 核心结构清晰，命名和注释可读性较好 |
| 可测试性 | 4 | 已有较完整单测与可分层测试的服务设计 |
| 可修改性 | 3 | 主要逻辑可改，但仍有一定耦合点 |
| 可移植性 | 2 | 运行环境与执行方式尚未标准化，跨环境风险较高 |
| 可重用性 | 2 | 业务规则与文案绑定较深，跨项目复用成本高 |

## 逐项说明

### 1) 可理解性（4 分）

- 优点：
  - 分层较明确：`models`、`engine`、`services`、`infrastructure`、`snapshot`。
  - 函数命名与中文提示文案一致，行为可追踪。
  - README 已覆盖基本玩法与测试执行。
- 扣分点：
  - `GameEngine` 仍承担较多编排与状态管理职责，阅读成本随功能增加而上升。

补充判断依据：
- 代码命名整体统一，模块分层已经形成稳定心智模型，新成员可快速定位职责。
- 关键业务路径（战斗、存读档、复活）已有服务拆分，降低了“单文件追踪到底”的阅读压力。
- 仍未满分的原因是引擎层在“流程编排 + 文案组织 + 兼容属性”上存在混合职责，理解时需要跨越更多上下文。

改进建议（非低分项，作为增量优化）：
- 将“结果文本渲染”逐步收口到适配层，进一步减轻引擎层认知负担。
- 为核心流程补一页“调用时序说明”（命令 -> 引擎 -> 服务 -> 仓储），缩短新人上手时间。

### 2) 可测试性（4 分）

- 优点：
  - 已有 `pytest` 测试体系与 CI 自动执行。
  - `CombatService`、`JsonSaveRepository` 可独立测试。
  - 关键路径（战斗、存读档、命令解析）已有回归测试。
- 扣分点：
  - 端到端场景测试和异常路径覆盖仍有提升空间。

补充判断依据：
- 已形成可执行的回归保护：服务层与引擎层均有针对关键分支的断言。
- CI 已纳入自动测试与复杂度检查，能对回归和结构退化提供第一道门禁。
- 扣分点集中在“跨模块业务流程”与“异常场景矩阵”尚未体系化，例如多步存读档 + 战斗状态联合回归。

改进建议（非低分项，作为增量优化）：
- 以用户故事为主线建立场景化测试清单，补齐跨模块链路用例。
- 为关键失败分支增加参数化测试，减少重复用例代码并提高覆盖密度。

### 3) 可修改性（3 分）

- 优点：
  - 模块边界初步形成，重构时可在服务层落刀。
  - 存档快照与运行时模型已区分，便于演进。
- 风险：
  - 文本输出与业务逻辑在部分方法中耦合，修改业务规则时可能连带改动大量字符串断言。

补充判断依据：
- 当前结构已从“单体引擎方法”向“服务化分工”演进，常见改动可在局部完成。
- 但在引擎层仍有部分兼容字段与流程控制耦合，导致改动影响面难以完全预测。
- 因此判定为 3 分：可持续演进，但需要持续抽离职责，避免再次回到上帝类。

建议优先级：
- P1：继续推进 InventoryService、LegacyCommandAdapter、WorldBuilder。
- P2：将输出文案和业务结果结构彻底解耦，降低断言脆弱性。

### 4) 可移植性（2 分）

- 主要问题：
  - 目前以“本地命令运行”为主，缺少标准化环境描述（如 Python 版本范围、虚拟环境约束、统一启动入口）。
  - 控制台字符与编码依赖终端能力，跨平台终端显示可能存在差异。
  - 依赖管理较轻量，缺少锁定策略，团队环境可能出现版本漂移。

#### 改进方案（低于 3 分，必须整改）

1. 环境标准化
- 新增 `.python-version` 或在 README 明确推荐版本（例如 3.11）。
- 统一虚拟环境与安装步骤：`python -m venv .venv`、激活、`pip install -r requirements.txt`。
- 验收标准：新成员在全新机器 30 分钟内完成启动与测试。

2. 运行入口标准化
- 在 README 固化“开发启动命令”和“测试命令”。
- 补充常见平台差异说明（Windows/Linux/macOS 的激活方式）。
- 验收标准：三平台按文档可执行成功。

3. 依赖可复现
- 在后续迭代引入依赖锁定文件（如 `requirements-lock.txt` 或 pip-tools）。
- 验收标准：CI 与本地安装依赖版本一致。

分阶段实施建议：
- 第 1 周：固化 Python 推荐版本与多平台启动说明，统一开发入口。
- 第 2 周：引入依赖锁定与升级策略（谁升级、何时升级、如何回滚）。
- 第 3 周：在 CI 增加环境一致性校验（版本校验/依赖漂移检测）。

### 5) 可重用性（2 分）

- 主要问题：
  - 领域规则、输出文案、命令交互耦合较深，游戏逻辑难直接迁移到其它前端或其它游戏主题。
  - `main.py` 中世界构建逻辑偏硬编码，缺少配置化加载。

#### 改进方案（低于 3 分，必须整改）

1. 抽离领域服务接口
- 将“战斗结算结果”与“显示文案”进一步分离：服务返回结构化结果，UI 层负责渲染文本。
- 验收标准：在不改服务层的前提下可切换一套新的输出文案。

2. 世界数据配置化
- 将房间、敌人、物品定义迁移到 JSON/YAML 配置，保留校验器。
- 验收标准：新增地图无需改 Python 代码，仅改配置即可运行。

3. 命令层适配抽象
- 保持 `CommandHandler` 作为适配器，并预留 API/GUI 输入通道。
- 验收标准：新增一种输入通道时复用 `GameEngine` 与服务层，不复制业务代码。

分阶段实施建议：
- 第 1 周：先定义稳定的服务返回结构（避免上层直接依赖文案）。
- 第 2 周：把地图和物品配置抽离为可加载数据，并加入配置校验。
- 第 3 周：增加一条非 CLI 的轻量输入通道（如 mock API）验证复用边界。

## 风险与收益优先级

| 因素 | 当前分数 | 风险级别 | 对交付影响 | 优先级 |
|------|----------|----------|------------|--------|
| 可移植性 | 2 | 高 | 新成员环境初始化失败、跨平台行为不一致 | P0 |
| 可重用性 | 2 | 高 | 新场景扩展成本高、重复开发概率高 | P0 |
| 可修改性 | 3 | 中 | 中期迭代中改动影响面扩大 | P1 |
| 可测试性 | 4 | 中低 | 回归可控，但仍有场景缺口 | P1 |
| 可理解性 | 4 | 低 | 新人可上手，后续需文档持续跟进 | P2 |

## 下一迭代验收建议

- 结构目标：完成 `CheckpointService`、`InventoryService`、`WorldBuilder` 三项职责边界稳定化。
- 测试目标：以用户故事为单位补齐跨模块回归路径，并保持 CI 全绿。
- 文档目标：README 与待办状态保持同频更新，确保“代码状态 = 文档状态”。

## 结论

当前项目在“可理解性、可测试性、可修改性”方面达到可迭代水平；“可移植性、可重用性”仍是主要短板，应作为下一轮重构与工程化改进重点。
