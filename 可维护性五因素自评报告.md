# 可维护性五因素自评报告

对照教材 §8.4.1 的五个因素，对当前代码（`game/` 与 `tests/`）进行 1-5 分制自评。

## 评分总览

| 因素 | 分数（1-5） | 结论 |
|------|-------------|------|
| 可理解性 | 4 | 核心结构清晰，命名和注释可读性较好 |
| 可测试性 | 4 | 已有较完整单测与可分层测试的服务设计 |
| 可修改性 | 3 | 主要逻辑可改，但仍有一定耦合点 |
| 可移植性 | 2 | 运行环境与执行方式尚未标准化，跨环境风险较高 |
| 可重用性 | 2 | 业务规则与文案绑定较深，跨项目复用成本高 |

## 逐项说明

### 1) 可理解性（4 分）

- 优点：
  - 分层较明确：`models`、`engine`、`services`、`infrastructure`、`snapshot`。
  - 函数命名与中文提示文案一致，行为可追踪。
  - README 已覆盖基本玩法与测试执行。
- 扣分点：
  - `GameEngine` 仍承担较多编排与状态管理职责，阅读成本随功能增加而上升。

### 2) 可测试性（4 分）

- 优点：
  - 已有 `pytest` 测试体系与 CI 自动执行。
  - `CombatService`、`JsonSaveRepository` 可独立测试。
  - 关键路径（战斗、存读档、命令解析）已有回归测试。
- 扣分点：
  - 端到端场景测试和异常路径覆盖仍有提升空间。

### 3) 可修改性（3 分）

- 优点：
  - 模块边界初步形成，重构时可在服务层落刀。
  - 存档快照与运行时模型已区分，便于演进。
- 风险：
  - 文本输出与业务逻辑在部分方法中耦合，修改业务规则时可能连带改动大量字符串断言。

### 4) 可移植性（2 分）

- 主要问题：
  - 目前以“本地命令运行”为主，缺少标准化环境描述（如 Python 版本范围、虚拟环境约束、统一启动入口）。
  - 控制台字符与编码依赖终端能力，跨平台终端显示可能存在差异。
  - 依赖管理较轻量，缺少锁定策略，团队环境可能出现版本漂移。

#### 改进方案（低于 3 分，必须整改）

1. 环境标准化
- 新增 `.python-version` 或在 README 明确推荐版本（例如 3.11）。
- 统一虚拟环境与安装步骤：`python -m venv .venv`、激活、`pip install -r requirements.txt`。
- 验收标准：新成员在全新机器 30 分钟内完成启动与测试。

2. 运行入口标准化
- 在 README 固化“开发启动命令”和“测试命令”。
- 补充常见平台差异说明（Windows/Linux/macOS 的激活方式）。
- 验收标准：三平台按文档可执行成功。

3. 依赖可复现
- 在后续迭代引入依赖锁定文件（如 `requirements-lock.txt` 或 pip-tools）。
- 验收标准：CI 与本地安装依赖版本一致。

### 5) 可重用性（2 分）

- 主要问题：
  - 领域规则、输出文案、命令交互耦合较深，游戏逻辑难直接迁移到其它前端或其它游戏主题。
  - `main.py` 中世界构建逻辑偏硬编码，缺少配置化加载。

#### 改进方案（低于 3 分，必须整改）

1. 抽离领域服务接口
- 将“战斗结算结果”与“显示文案”进一步分离：服务返回结构化结果，UI 层负责渲染文本。
- 验收标准：在不改服务层的前提下可切换一套新的输出文案。

2. 世界数据配置化
- 将房间、敌人、物品定义迁移到 JSON/YAML 配置，保留校验器。
- 验收标准：新增地图无需改 Python 代码，仅改配置即可运行。

3. 命令层适配抽象
- 保持 `CommandHandler` 作为适配器，并预留 API/GUI 输入通道。
- 验收标准：新增一种输入通道时复用 `GameEngine` 与服务层，不复制业务代码。

## 结论

当前项目在“可理解性、可测试性、可修改性”方面达到可迭代水平；“可移植性、可重用性”仍是主要短板，应作为下一轮重构与工程化改进重点。
