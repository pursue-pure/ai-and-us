# 可维护性五因素自评报告 (Sprint 2)

本报告对照教材 §8.4.1 的五个维度，对 AI and Us 文字冒险游戏项目当前代码进行 1-5 分制自评。

## 1. 维度自评与分析

| 维度 | 自评分 | 评价分析 |
| :--- | :---: | :--- |
| 可理解性 (Understandability) | 4 | 命名规范（遵循 PEP8），模块职责清晰。采用分层设计，将实体（Models）、业务逻辑（Services）和基础设施（Infrastructure）做了显式隔离。代码逻辑直观，核心流程（战斗、移动、存读档）易于追踪。 |
| 可测试性 (Testability) | 4 | 已建立完善的 tests 目录结构。通过依赖注入（如 GameEngine 组合 Repository 与 Service）提升了可测性。CombatService 等核心逻辑可独立单测，回归成本较低。 |
| 可修改性 (Modifiability) | 4 | 采用仓储模式（Repository Pattern）和服务拆分思路（Service）。新增存储方式或调整战斗规则时，通常只需局部修改，整体波及效应较低。 |
| 可移植性 (Portability) | 3 | 代码基于 Python 3.10+，依赖少，易于在 Windows/Linux/macOS 运行。但当前仍以本地文件系统（JSON）为主要持久化方式，迁移至云原生环境（如对象存储或 NoSQL）需要额外适配。 |
| 可重用性 (Reusability) | 2 | 低于 3 分。虽然 CombatService 与 Models 具备一定通用性，但目前仍存在与当前题材绑定较深的数值和文本硬编码。缺乏统一配置加载机制，跨项目复用成本较高。 |

## 2. 低于 3 分因素的改进方案

### 可重用性 (Reusability) - 改进方案

1. 数据与逻辑分离
- 将写死在 models.py 和 engine.py 中的房间描述、怪物属性、物品效果提取到外部配置。
- 增加 AssetLoader，负责从配置动态实例化游戏对象。
- 增加配置合法性校验（字段完整性、类型校验、边界值校验），避免配置错误直接进入运行时。
- 验收标准：新增地图元素时，仅修改配置即可生效，不需要修改核心业务代码。

2. 泛化服务接口
- 为 CombatService 定义更通用的接口（如 ICombatSystem），以支持不同战斗规则（回合制、变体规则等）。
- 保留当前战斗逻辑作为默认实现，允许通过注入方式替换规则。
- 验收标准：切换规则实现时不改动引擎主流程，且原有回归测试持续通过。

3. 插件化指令系统
- 将 commands.py 逐步改造成命令模式（Command Pattern），通过注册机制扩展新指令，避免频繁修改核心引擎。
- 保留对现有命令格式的兼容（如简写命令和历史输入形式），降低迁移风险。
- 验收标准：新增命令仅需新增实现并注册，不需要修改主分发逻辑。

4. 结果与文案解耦
- 服务层返回结构化业务结果，命令层负责渲染文本输出，减少业务规则与展示文案耦合。
- 验收标准：替换一套输出文案时，不需要修改服务层核心逻辑。

5. 世界构建配置化
- 将 main.py 中房间、敌人、物品的初始化逻辑逐步迁移到配置文件与加载器。
- 验收标准：新增一个场景主题时，主要改动集中在配置文件和资源层。

## 3. 改进建议（可移植性）

1. 固化运行环境
- 在文档中明确推荐 Python 版本与跨平台虚拟环境激活方式，降低新成员环境差异。
- 明确统一的启动命令和测试命令，减少环境差异导致的执行歧义。
- 验收标准：新成员在全新机器 30 分钟内完成安装、启动和测试执行。

2. 提升依赖可复现性
- 后续可引入锁定依赖文件，减少本地与 CI 版本漂移。
- 建议在依赖升级时采用固定流程（升级、回归、回滚预案），避免隐性破坏。
- 验收标准：CI 与本地安装依赖版本一致，升级后关键回归测试无新增失败。

3. 保持存储层可替换
- 维持 Repository 抽象边界，后续新增数据库或云存储实现时不影响上层业务逻辑。
- 先保留 JSON 实现作为基线，再新增对象存储或数据库实现并通过同一接口接入。
- 验收标准：新增一种存储实现时，业务层调用不发生改动。

4. 跨平台编码与路径一致性
- 统一文件编码和路径处理策略，避免不同终端和系统下的读取差异。
- 验收标准：在不同终端上执行同一套回归测试结果一致。
