# 可维护性五因素自评报告 (Sprint 2)

本报告对照教材 §8.4.1 的五个维度，对“AI and Us”文字冒险游戏项目的当前代码进行 1-5 分制自评。

## 1. 维度自评与分析

| 维度 | 自评分 | 评价分析 |
| :--- | :---: | :--- |
| **可理解性 (Understandability)** | **4** | 命名规范（遵循 PEP8），模块职责清晰。采用了领域驱动设计（DDD）的思想，将实体（Models）、业务逻辑（Services）和基础设施（Infrastructure）进行了显式隔离。代码逻辑直观，核心流程（如战斗、移动）易于追踪。 |
| **可测试性 (Testability)** | **4** | 建立了完善的 `tests/` 目录结构。由于采用了依赖注入（如 `GameEngine` 注入 `Repository` 和 `Service`），核心组件可以方便地使用 Mock 对象进行单元测试。战斗逻辑（CombatService）完全解耦，可独立于引擎运行测试。 |
| **可修改性 (Modifiability)** | **4** | 采用了仓储模式（Repository Pattern）和策略模式（Service），新增一种存储方式（如数据库）或修改战斗算法只需更改特定的类或模块，不会引发整体代码的连锁反应（波及效应低）。 |
| **可移植性 (Portability)** | **3** | 代码基于 Python 3.10+ 标准库，依赖项极少（仅 pytest），易于在 Windows/Linux/macOS 运行。但目前强绑定了本地文件系统（JSON 存储），若要迁移到云原生环境（如 S3 或 NoSQL），需要重写基础设施层。 |
| **可重用性 (Reusability)** | **2** | **低于 3 分**。虽然 `CombatService` 和 `Models` 具有一定的通用性，但目前代码中存在较多与特定游戏数值和文本硬编码相关的逻辑。缺乏通用的配置加载机制，导致无法轻量化地复用到其他题材的文字冒险游戏中。 |

## 2. 低于 3 分因素的改进方案

### 可重用性 (Reusability) - 改进方案：
1.  **数据与逻辑分离**：
    *   将硬编码在 `models.py` 和 `engine.py` 中的房间描述、怪物属性数值、物品效果提取到外部配置文件（如 `data/rooms.yaml` 或 `data/config.json`）。
    *   实现一个 `AssetLoader` 类，负责从配置中动态实例化游戏对象。
2.  **泛化服务接口**：
    *   为 `CombatService` 定义通用的 `ICombatSystem` 接口，以便支持不同规则的战斗系统（如回合制 vs. 即时制）。
3.  **插件化指令系统**：
    *   将 `commands.py` 改造为命令模式（Command Pattern），允许通过注册机制动态添加新的游戏指令，而无需修改核心引擎代码。
